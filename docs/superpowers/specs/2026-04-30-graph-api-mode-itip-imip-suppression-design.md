# Graph API Mode — Suppress Nextcloud iTip / iMip on Synced Calendars

- **Date:** 2026-04-30
- **Author:** L. Pasmans (Sendent)
- **Status:** Draft for review
- **Target Nextcloud versions:** 28 – 33

## Background

Sendent Sync bi-directionally synchronises a user's Microsoft Exchange calendar with a Nextcloud calendar. When a synced user creates, updates, or cancels an event with attendees:

- **Microsoft side** — when the event is pushed to Exchange, Exchange/Graph dispatches meeting invitations to all attendees automatically.
- **Nextcloud side** — Sabre/DAV's Schedule plugin generates iTip messages, and `OCA\DAV\CalDAV\Schedule\IMipPlugin` ships those as iMip emails to external attendees, *and* delivers iTip locally to other Nextcloud users on the same instance.

The result: every attendee receives **two** invitations (one from Exchange, one from Nextcloud), and any internal Nextcloud attendee gets a local CalDAV drop that can then itself be pushed back to Exchange — risking loops and duplicates.

Until now Sendent Sync used the EWS `SendMeetingInvitations = SendToNone` option to keep Exchange silent and let Nextcloud's iMip handle delivery. **EWS support for Exchange Online ends October 2026**, and Microsoft Graph has no equivalent suppression flag. The fix flips the model: let Exchange/Graph send invitations and silence Nextcloud's scheduling pipeline for synced calendars.

## Goals

When the new admin toggle ("Graph API mode") is **enabled** and a CalDAV scheduling event is generated by an active Sendent user against their synced calendar, suppress *all* iTip processing for that event:

- No outbound iMip emails to external attendees
- No internal CalDAV iTip delivery to other Nextcloud users on the same instance
- Exchange/Graph becomes the sole authority for scheduling on synced calendars

When the toggle is **disabled** (the default) the app behaves exactly as it does today — no scheduling pipeline interception happens.

## Non-Goals

- No new personal-settings (per-user) toggle. Always-on per active Sendent user once the admin toggle is enabled.
- No handling of *incoming* iTip replies. Exchange handles attendee responses via the user's Exchange mailbox; Nextcloud never sees them on synced calendars.
- No change to non-CalDAV mail (notifications, share emails, internal NC notifications are all unaffected).
- No retroactive cleanup of duplicates already sent before the toggle is flipped.
- No support for multiple synced calendars per user in this iteration. The design must leave room to expand to a set later, but `SyncUser.calendar` currently stores a single calendar URI and that is what we match against.

## Suppression Conditions (all must hold)

1. App config `graphApiMode === 'true'`
2. The CalDAV request is from an authenticated Nextcloud user
3. That user has an `active = true` row in the `SyncUser` table
4. The CalDAV request URI's calendar segment equals the user's `SyncUser.calendar`

If any condition fails, no interception happens — Nextcloud's normal scheduling pipeline runs untouched.

## Approach

Hook Sabre/DAV's `schedule` event with a listener registered at a **higher priority** (i.e. lower numeric priority value, since Sabre invokes lowest-priority listeners first) than `OCA\DAV\CalDAV\Schedule\IMipPlugin`. For each iTip message we evaluate the suppression conditions; on match we set `$iTipMessage->scheduleStatus = '1.1'` ("Message delivered"), causing every downstream scheduling listener — including the iMip mailer and the internal-delivery listener — to skip the message.

This is the same Sabre extension contract that any third-party scheduling plugin uses, so it is stable across Nextcloud 28–33 and survives upgrades. No core overrides, no decoration of `IMipPlugin`, no service-container patches.

### Why not the alternatives

- **Decorate `IMipPlugin` via the DI container** — Nextcloud constructs DAV plugins through `OCA\DAV\Server`, not always via the service container, so the override may not even apply. Even where it does, internals can change between NC 28 and 33. Fragile.
- **Listen on `CalendarObjectUpdatedEvent`** — fires after the schedule pipeline has already run; iMip emails are already queued. Non-starter.
- **Disable Nextcloud's mail transport globally** — far too broad; breaks unrelated app emails.

### Why a Sabre plugin and not a plain listener

Sabre's `Server` event emitter is a different system from Nextcloud's `IEventDispatcher`. The cleanest way to attach a listener to Sabre's `schedule` event is to provide a Sabre `ServerPlugin` whose `initialize($server)` method calls `$server->on('schedule', $callback, $priority)`. Nextcloud exposes `SabrePluginAddPluginEvent` precisely for this hand-off: when the DAV server is being constructed, the event fires, we receive the `Server` instance, and we add our plugin.

## Data Flow

1. User saves an event with attendees on their synced calendar (CalDAV `PUT` to `/calendars/<uid>/<calendar>/<event>.ics`).
2. The Sabre Schedule plugin processes the change and generates one or more `Sabre\VObject\ITip\Message` objects (one per attendee or organiser depending on the change).
3. For each iTip message, Sabre emits the `schedule` event.
4. **`SchedulingSuppressorPlugin`'s listener runs first** (priority lower than IMipPlugin's):
   - Asks `SchedulingSuppressionService::shouldSuppress($iTipMessage, $server)`.
   - If `true`, sets `$iTipMessage->scheduleStatus = '1.1; suppressed by Sendent Sync (Graph API mode)'` and stops iterating handlers (`return false` if needed by the Sabre contract — see Open Questions).
5. `IMipPlugin` and the internal-delivery listener run next; both check `$iTipMessage->scheduleStatus` and skip when it is already set. No email is sent, no internal iTip drop occurs.
6. The CalDAV write itself proceeds normally — the event is stored on the calendar as usual. Sendent Sync's existing pipeline subsequently pushes it to Exchange, where Exchange/Graph handles all attendee notification.

### `SchedulingSuppressionService::shouldSuppress` logic

```
1. If appConfig('graphApiMode') !== 'true': return false
2. uid = current authenticated user (IUserSession)
   If null: return false
3. syncUser = SyncUserMapper::findByUid(uid)
   If null OR syncUser.active !== true: return false
4. requestPath = $server->getRequestUri()
   Expected shape: 'calendars/<uid>/<calendar-uri>/<event>.ics'
   - If the path does not start with 'calendars/': return false (e.g. principals/...)
   - If fewer than three segments after the 'calendars/' prefix: return false
   - Extract <calendar-uri> as the segment immediately after <uid>
   On any extraction failure, fail closed: return false (never suppress on ambiguity, so the user receives invitations from at least one source)
5. If <calendar-uri> !== syncUser.calendar: return false
6. Return true
```

The "fail closed" stance in step 4 means: if for any reason we cannot determine the calendar from the request, we let Nextcloud's normal pipeline run rather than risk silently dropping invitations.

## Configuration

| Key | Default | Type | Where |
| --- | --- | --- | --- |
| `graphApiMode` | `'false'` | string `'true'`/`'false'` | App config, set via admin settings |

The toggle is hidden until the admin actively enables it. Hint copy on the admin page:

> **Graph API mode** — Enable when Sendent Sync uses Microsoft Graph (rather than Exchange Web Services) for Exchange access. While enabled, Nextcloud will not send meeting invitations or process internal scheduling messages for calendars synced by Sendent Sync; Exchange/Graph becomes the sole sender of meeting invitations. Leave disabled if Sendent Sync is using EWS, otherwise attendees may not receive invitations.

## Testing

### Unit tests

- `SchedulingSuppressionServiceTest`:
  - Returns `false` when `graphApiMode` is disabled
  - Returns `false` when no authenticated user
  - Returns `false` when `SyncUser` row missing
  - Returns `false` when `SyncUser.active = false`
  - Returns `false` when request calendar URI does not match `SyncUser.calendar`
  - Returns `false` when the request URI cannot be parsed (fail closed)
  - Returns `true` when all four conditions hold

### Integration / manual tests

| Scenario | Expectation |
| --- | --- |
| Graph API mode OFF, Sendent user creates event with external attendee on synced calendar | iMip email sent (current behaviour) |
| Graph API mode ON, same scenario | No iMip email sent; event stored on calendar; Sendent Sync pushes to Exchange |

| Graph API mode ON, Sendent user cancels an event (METHOD:CANCEL) | No CANCEL iMip / iTip dispatched from Nextcloud |
| Toggle flipped ON, then OFF, then ON again | Behaviour follows the current setting on each subsequent CalDAV write |

### Compatibility verification

Confirm against both Nextcloud 28 (minimum) and Nextcloud 33 (maximum):

- `OCA\DAV\Events\SabrePluginAddPluginEvent` exists and fires
- `OCA\DAV\CalDAV\Schedule\IMipPlugin` priority on the `schedule` event has not changed in a way that breaks our higher-priority hook
- Setting `scheduleStatus` to `'1.1'` on an iTip message causes IMipPlugin and internal-delivery listeners to skip

If the priority value changes between versions, the design still holds — we just need a version-aware constant. Document the observed priority for both NC 28 and NC 33 in the implementation plan.

## Risks & Open Questions

1. **Sabre listener priority semantics** — Sabre/Event invokes the lowest-numeric-priority listener first. We need to register at a value below IMipPlugin's. The plan should determine the exact constant by reading `OCA\DAV\CalDAV\Schedule\IMipPlugin::initialize()` in both NC 28 and NC 33.
2. **Multiple iTip listeners** — confirm there is no other Nextcloud listener registered above IMipPlugin that could deliver iTip before ours runs. The plan should map all `schedule`-event handlers in the supported NC range.
3. **Resolving sender → uid** — preferred path is `IUserSession::getUser()` (the request is authenticated). Fallback if that proves unreliable in the DAV context: parse the principal URI from `$iTipMessage->sender` (`mailto:`) and look it up via `IUserManager::getByEmail()`.
4. **`SyncUser.calendar` storage format** — verify the field stores a URI segment matching the third segment of the CalDAV request path, not a display name or numeric id. If formats differ, the comparison logic needs a normalisation step.
5. **Future multi-calendar support** — when we extend to multiple synced calendars per user, replace the equality check in step 5 with set membership. No other part of the design changes.
6. **`return false` on the Sabre `schedule` event** — Sabre's emitter convention is that returning `false` from a listener stops further propagation. We should rely on `scheduleStatus` (the documented contract) rather than stopping propagation, so other custom listeners on the host can still observe the message.

## Out of scope (deferred)

- Personal-settings toggle (admin-level only for v1)
- Per-calendar selection inside the admin UI (today the synced calendar is implicit per user)
- Telemetry / counters for suppressed messages (could be added later for support diagnostics)
